# Implementation Plan — AI-Generated Initialization Sequence + Variable Interpolation

Status: draft
Author: scenario clone session, 2026-04-27
Targets: `backend/app/`, `frontend/src/`

---

## Why

Two related capabilities are missing from Project Lunar today:

1. **Variable interpolation in `opening_narrative`** (and adjacent fields). The
   `SetupWizard` already interpolates `{var_name}` inside question prompts
   (`frontend/src/components/SetupWizard.jsx:4`), but the opening narrative
   shown by `GameCanvas.jsx:370` is rendered literally:
   `<ReactMarkdown>{activeScenario.opening_narrative}</ReactMarkdown>`.
   Result: a scenario author who writes `You are {character_name}, a {race}…`
   sees the placeholders verbatim in-game (we hit this with the One Piece
   import). Setup answers do reach the narrator via the `CHARACTER SETUP`
   block in `backend/app/services/game_session.py:383`, so the LLM knows the
   character — but the static opening doesn't.

2. **Optional AI-generated opening per campaign.** Today the opening is one
   fixed string in `scenarios.opening_narrative`. We want an opt-in mode where
   the opening is freshly generated by the narrator LLM at campaign start,
   using the player's setup answers + scenario tone/lore. This produces a
   one-of-a-kind hook for each playthrough without asking the author to write
   conditional templates.

The two features are complementary: interpolation is the "cheap and
deterministic" path, AI generation is the "rich but stochastic" path. Authors
should be able to pick per scenario.

## Feature 1 — Variable interpolation

### 1.1 Interpolation rules

Match the rules already implemented for the wizard prompts (so authors learn
one syntax):

- Token shape: `{var_name}` where `var_name` matches `^[a-z][a-z0-9_]*$`.
- The `var_name` is looked up in the campaign's `setup_answers` map; the
  resolved string is `answer.value` (unmodified, no smart-casing).
- **Missing var** → leave the original `{var_name}` literal in place AND
  log a single warning per (campaign_id, var_name) pair to aid debugging.
  Rationale: silent empty substitutions hide author typos; visible literals
  surface them on first playthrough.
- **Literal braces** → support `{{` and `}}` as escapes (becomes `{` / `}`),
  matching Python's `str.format` convention. Authors writing JSON/code
  fragments inside lore won't get mangled.
- Replacement is single-pass — substituted values are NOT re-interpolated
  (otherwise an answer like `{my favorite weapon: {weapon}}` would loop).

### 1.2 Where interpolation runs

| Field | Interpolated when | Why |
|---|---|---|
| `opening_narrative` | served via `/scenario-view`; seeded into narrator history | what the player sees; what the narrator anchors on |
| `tone_instructions` | when `game_session._build_system_prompt()` assembles the system message | author can reference player attributes in voice rules |
| `lore_text` | when injected into the narrator's system prompt | same |
| `description` | not interpolated | shown only in the home archive — pre-campaign, no answers exist |
| `setup_questions[*].prompt` | already interpolated by the frontend wizard | unchanged |

The narrator's `CHARACTER SETUP` block (`game_session.py:383`) keeps working
unchanged — interpolation is **additive**, not a replacement.

### 1.3 Files to add / modify

**New** — `backend/app/services/scenario_interpolation.py`

```python
import logging
import re

_VAR_RE = re.compile(r"\{([a-z_][a-z0-9_]*)\}")
_logger = logging.getLogger(__name__)


def interpolate(template: str, answers: dict, *, context: str = "") -> str:
    """Substitute {var_name} tokens with values from setup_answers.

    answers: the dict shape produced by SetupWizard — values are dicts with
             at least {'value': str} (frontend builds them in buildAnswer()).

    Missing variables are left literal so authors can spot typos.
    `{{` / `}}` are honored as escapes for literal braces.
    """
    if not template:
        return ""
    # Two-step to honor {{ }} escapes without disturbing the var regex.
    OPEN, CLOSE = "OPEN", "CLOSE"
    template = template.replace("{{", OPEN).replace("}}", CLOSE)

    missing: set[str] = set()
    def _resolve(match: re.Match) -> str:
        key = match.group(1)
        a = answers.get(key)
        if isinstance(a, dict):
            value = a.get("value")
        else:
            value = a
        if not value:
            missing.add(key)
            return match.group(0)
        return str(value)

    out = _VAR_RE.sub(_resolve, template)
    out = out.replace(OPEN, "{").replace(CLOSE, "}")

    if missing and context:
        _logger.warning(
            "Unresolved interpolation tokens in %s: %s",
            context, sorted(missing),
        )
    return out
```

```python
@router.get("/{campaign_id}/scenario-view")
def get_scenario_view(campaign_id: str):
    with ScenarioStore(_SCENARIO_DB_PATH) as store:
        campaign = store.get_campaign(campaign_id)
        if not campaign:
            raise HTTPException(status_code=404)
        scenario = store.get_scenario(campaign.scenario_id)

    answers = campaign.setup_answers or {}
    opening = (
        campaign.generated_opening
        or interpolate(scenario.opening_narrative, answers,
                       context=f"campaign:{campaign_id}:opening")
    )
    return {
        "title": scenario.title,
        "language": scenario.language,
        "opening_narrative": opening,
        "tone_instructions": interpolate(scenario.tone_instructions, answers,
                                         context=f"campaign:{campaign_id}:tone"),
        "lore_text": interpolate(scenario.lore_text, answers,
                                 context=f"campaign:{campaign_id}:lore"),
        "opening_mode": scenario.opening_mode,
    }
```

### 1.4 Edge cases

- **Empty `setup_answers`** (campaign hasn't completed setup): the view
  endpoint returns the template with all literals visible. Acceptable
  because `PlayGate` redirects to wizard before showing GameCanvas anyway.
- **Answers with non-string values** (e.g. `null` from a skipped optional):
  `interpolate()` treats falsy as missing → literal stays. No crash.
- **Recursive references** (`{a}` resolves to `"{b}"`): not re-expanded.
  Documented as intentional; if authors need this, they can just inline.
- **Markdown collisions**: `{` is not special in commonmark, so we don't
  break `<ReactMarkdown>`. Code blocks still render literal `{name}`
  because we interpolate **before** markdown — to make code blocks safe,
  honor a `\{name}` backslash escape: a `\` immediately before `{` skips
  the substitution. Implementation: pre-pass the regex with negative
  look-behind `(?<!\\)\{...}`. Trim the leading `\` after substitution.

## Feature 2 — AI-generated Initialization Sequence

### 2.1 UX flow

**Author side (ScenarioBuilder)**:

1. Add a `Initialization Sequence Mode` toggle:
   - `Fixed Text` (current behavior)
   - `Generated by AI`
2. When `AI` is selected:
   - The existing `opening_narrative` textarea is repurposed as an optional
     **director's note** ("guidance the AI must respect; leave blank for a
     default atmospheric opening"). Stored as a new field
     `ai_opening_directive` for clarity, NOT in `opening_narrative`.
   - A "Preview AI Opening" button calls
     `POST /api/scenarios/preview-opening` with mock answers (we synthesize
     one example value per question — first option's label, or var_name as
     placeholder for text). Result is shown read-only beneath the button.

**Player side (campaign creation)**:

1. Setup wizard runs as today.
2. After the wizard's final `POST /setup-answers`, the backend triggers
   opening generation if `scenario.opening_mode == 'ai'`. The generated
   text is persisted in `campaigns.generated_opening`.
3. GameCanvas pulls `generated_opening` via `/scenario-view` (Feature 1
   already routes this — `generated_opening` takes precedence over
   interpolated `opening_narrative`).
4. The "Re-roll opening" affordance (a small button visible only when no
   actions have been taken yet) calls
   `POST /api/game/{cid}/regenerate-opening` for one retry. After the first
   action we lock it (re-rolling the opening mid-story breaks history).

### 2.3 New module — `backend/app/engines/opening_generator.py`

```python
from app.engines.llm_router import LLMRouter, LLMConfig
from app.services.scenario_interpolation import interpolate

_SYSTEM = """You write the cold-open of an interactive narrative RPG.

Constraints:
- Second person ("You ...").
- 180-360 words, 4-8 short paragraphs.
- Honor the SCENARIO TONE rules and the world LORE.
- Establish setting, sensory detail, and one immediate dramatic hook.
- Reference the player's CHARACTER SETUP organically — never list it as
  a stat block.
- End on a question, beat, or speaker-tag that invites the player's first
  action; do not narrate the player's response.
- No headings, no markdown beyond paragraph breaks.
"""


async def generate_opening(
    *, language: str, tone: str, lore: str,
    character_setup_lines: list[str],
    director_note: str = "",
    router: LLMRouter,
) -> str:
    user = (
        f"LANGUAGE: write the entire opening in {language}.\n\n"
        f"SCENARIO TONE:\n{tone[:6000]}\n\n"
        f"WORLD LORE:\n{lore[:6000]}\n\n"
        f"CHARACTER SETUP:\n" + "\n".join(character_setup_lines) + "\n\n"
        + (f"DIRECTOR'S NOTE:\n{director_note}\n\n" if director_note else "")
        + "Write the cold-open now."
    )
    messages = [
        {"role": "system", "content": _SYSTEM},
        {"role": "user", "content": user},
    ]
    return (await router.complete(messages, max_tokens=900)).strip()
```

The 6000-char truncation on tone/lore is a guardrail to stay within the
default `complete()` budget; the existing token-budget util can replace it
once the dynamic-budget refactor lands (see `memory/project_dynamic_context_budget.md`).

### 2.4 Endpoint changes

**`backend/app/api/routes_scenarios.py`**

- Extend `CreateScenarioRequest`:
  ```python

  ai_opening_directive: str = Field(default="", max_length=4000)
  ```
- New endpoint:
  ```python
  @router.post("/preview-opening")
  async def preview_opening(req: PreviewOpeningRequest):
      # synthesize sample answers from req.setup_questions if not provided
      lines = _format_setup_lines(req.sample_answers, req.setup_questions)
      router = LLMRouter(_default_llm_config())
      text = await generate_opening(
          language=req.language, tone=req.tone, lore=req.lore,
          character_setup_lines=lines,
          director_note=req.directive, router=router,
      )
      return {"opening": text}
  ```
- Export endpoint adds the two new scenario fields to the payload (so
  exported JSON round-trips faithfully).

**`backend/app/api/routes_game.py`**

- Modify `save_setup_answers`: after `update_setup_answers` succeeds,
  branch on `scenario.opening_mode`:

  ```python
  if scenario.opening_mode == "ai":
      lines = _format_setup_lines(payload, scenario.setup_questions)
      router = LLMRouter(_default_llm_config())
      text = await generate_opening(
          language=scenario.language, tone=scenario.tone_instructions,
  ```

  - 409 if any actions exist for the campaign (`event_store.get_all(...)`

  - Otherwise re-runs generation and updates `generated_opening`.

### 2.6 Telemetry

Each AI generation logs an `EventType.AI_OPENING_GENERATED` event with
payload `{ "char_count": N, "model": "...", "duration_ms": M }`. This gives
us a way to spot regressions in tone/length over time without surfacing it
to players.

### 2.7 Edge cases

- LLM failure during `save_setup_answers` → log, continue, fall back to
  the (interpolated) static opening. Player still gets to play.
- Re-roll requested after the first action → 409 with explanation.
- Author selects `mode='ai'` but writes no tone/lore → generator works
  but produces blander output; we surface this in the Preview panel as a
  hint ("Tip: richer tone/lore yields stronger openings").
- Existing scenarios → continue with `opening_mode='fixed'` (default).
  No data migration of content needed.

## Build sequence

1. **Migration v3** in `scenario_store.py` + dataclass updates +
   `update_generated_opening()` helper.
2. **`scenario_interpolation.py` + tests** (Feature 1, no LLM dependency).
3. Wire interpolation into `game_session.py` (tone/lore/opening seed) and
   the new `/scenario-view` endpoint.

   `/scenario-view`. SetupWizard refactored to import from the shared util.
5. **`opening_generator.py`** module + `/scenarios/preview-opening`
   endpoint + author-side UI.
6. **Player-side AI generation** — `save_setup_answers` triggers
   generation, `generated_opening` takes precedence in `/scenario-view`.
7. **Re-roll endpoint + UI**.
8. **Export/import round-trip** for the new scenario fields.
9. **E2E Playwright tests** for both modes.

Phases 1–4 are independently shippable and unblock the One Piece-style
scenarios that triggered this work. Phases 5–7 add the AI mode on top
without disturbing fixed-mode scenarios.

## Out of scope (intentionally)

- Per-question AI-generated answers (a different feature: agent-driven
  setup wizard).
- Mid-campaign opening re-writes (we explicitly lock after first action).
- Localized template variants per language (a single template can be
  multilingual via the LLM if `opening_mode='ai'`; static templates stay
  monolingual for now).
- Vector-similarity recall against story cards inside the opening prompt
  (deferred until the dynamic-budget RAG work in
  `memory/project_dynamic_context_budget.md` lands).
